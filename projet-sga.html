Semaine 4 · Partie 3 — Mise en pratique

Projet réel — Système de Gestion Académique

Mettre en application l'ensemble du cheminement : UML complet, schéma relationnel, optimisation MySQL.

Projet fil rouge de la formation : un Système de Gestion Académique (SGA), construit de bout en bout — de l'UML au SQL optimisé.

01 Pourquoi ce projet

Contexte volontairement choisi

C'est le propre contexte des étudiants (l'université elle-même). Il couvre naturellement une association N–N (étudiant–cours), un héritage possible (utilisateur → étudiant/enseignant), un cycle de vie clair (statut d'inscription), et assez de volumétrie pour justifier l'indexation et l'optimisation.

02 Périmètre fonctionnel

  • Un étudiant peut s'inscrire à plusieurs cours (dans la limite des places disponibles).
  • Un enseignant saisit les notes de ses cours.
  • Le système calcule automatiquement les moyennes.
  • Une inscription suit un cycle de vie : en attente → validée → terminée / annulée.

04 Script de départ — structure des tables

CREATE TABLE utilisateur (
  id INT AUTO_INCREMENT PRIMARY KEY,
  nom VARCHAR(100) NOT NULL,
  email VARCHAR(150) UNIQUE NOT NULL,
  role ENUM('etudiant','enseignant') NOT NULL
);

CREATE TABLE cours (
  id INT AUTO_INCREMENT PRIMARY KEY,
  titre VARCHAR(150) NOT NULL,
  credits INT NOT NULL,
  places_restantes INT NOT NULL,
  enseignant_id INT,
  FOREIGN KEY (enseignant_id) REFERENCES utilisateur(id)
);

CREATE TABLE inscription (
  id INT AUTO_INCREMENT PRIMARY KEY,
  etudiant_id INT NOT NULL,
  cours_id INT NOT NULL,
  statut ENUM('en_attente','validee','terminee','annulee') DEFAULT 'en_attente',
  date_inscription DATETIME DEFAULT CURRENT_TIMESTAMP,
  FOREIGN KEY (etudiant_id) REFERENCES utilisateur(id),
  FOREIGN KEY (cours_id) REFERENCES cours(id),
  UNIQUE KEY uniq_inscription (etudiant_id, cours_id)
);

CREATE TABLE note (
  id INT AUTO_INCREMENT PRIMARY KEY,
  inscription_id INT NOT NULL,
  valeur DECIMAL(4,2) NOT NULL,
  FOREIGN KEY (inscription_id) REFERENCES inscription(id)
);

05 Critères d'évaluation

  • Cohérence entre le diagramme de classes et le schéma SQL final.
  • Justification écrite de chaque index (quelle requête il accélère, pourquoi).
  • Transaction correctement utilisée sur l'opération d'inscription (vérification places + insertion atomique).
  • Preuve mesurable d'optimisation (temps d'exécution ou plan EXPLAIN avant/après).
