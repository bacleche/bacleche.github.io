Semaine 2 · Partie 1 — Conception UML

Diagrammes d'états & de déploiement

Cycle de vie des objets, puis architecture physique du système — la transition vers la conception architecturale.

Derniers diagrammes du cheminement — on referme la conception objet (états) et on bascule vers la conception architecturale (déploiement).

01 Diagramme d'états-transitions

Pourquoi il est nécessaire

Indispensable dès qu'un objet possède un statut qui évolue dans le temps selon des événements précis. Un seul diagramme par entité qui a un cycle de vie clair — pas besoin d'en faire un pour chaque classe.

Éléments

ÉlémentNotation
ÉtatRectangle à coins arrondis
TransitionFlèche étiquetée par l'événement déclencheur
État initial / finalCercle plein / cercle bordé

Exemple — PlantUML

@startuml
[*] --> EnAttente
EnAttente --> Validee : validerInscription()
EnAttente --> Annulee : annuler()
Validee --> Terminee : cloreSemestre()
Validee --> Annulee : annuler()
Terminee --> [*]
Annulee --> [*]
@enduml

02 Diagramme de déploiement — la conception architecturale

Pourquoi il est nécessaire

C'est ici qu'on quitte la conception objet pour l'architecture physique : où tourne quoi, et comment les composants communiquent. C'est le pont direct vers l'infrastructure réelle — exactement ce que vous configurez déjà quand vous déployez sur Vercel, Railway ou o2switch.

Éléments

ÉlémentNotationExemple concret
NœudCube 3DServeur, conteneur, poste client
ArtefactRectangle avec icône documentL'application qui tourne sur le nœud
Lien de communicationTrait entre nœudsProtocole utilisé (HTTPS, TCP)

Exemple — architecture d'un projet réel

@startuml
node "Vercel" {
  artifact "Frontend Next.js"
}
node "Railway" {
  artifact "Backend Django/DRF"
}
node "PostgreSQL (Railway)" {
  artifact "Base de données"
}
"Frontend Next.js" --> "Backend Django/DRF" : HTTPS / REST API
"Backend Django/DRF" --> "Base de données" : SQL
@enduml
Lien avec votre pratique Ce diagramme, c'est littéralement le schéma d'architecture de PRESF ARCHIVIS ou LogiTrack. Les étudiants qui verront ce diagramme comprendront immédiatement pourquoi le frontend et le backend sont déployés séparément.
