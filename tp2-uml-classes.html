Semaine 1 · Partie 1 — Conception UML

Diagramme de classes

Le diagramme pivot : structure du domaine et futur schéma de base de données.

Le diagramme le plus important du cours. C'est celui qui se transforme directement en schéma de base de données — le pont entre conception objet et implémentation SQL.

01 Pourquoi il est central

Le problème qu'il résout

Il fige la structure statique du domaine : quelles entités existent, quels attributs elles portent, comment elles se relient. Toute la partie 2 du cours (bases de données avancées) part de ce diagramme.

02 Anatomie d'une classe

+----------------------------+
|          Etudiant          |   <- nom de la classe
+----------------------------+
| - id: int                  |   <- attributs (- privé, + public)
| - nom: string               |
| - email: string             |
+----------------------------+
| + sInscrire(cours: Cours)  |   <- méthodes
+----------------------------+

04 Méthode de construction

  1. Repérer les noms communs importants du cahier des charges → candidats classes.
  2. Pour chaque classe, lister ses attributs propres (pas ceux d'une autre classe).
  3. Relier les classes entre elles et poser les cardinalités des deux côtés.
  4. Ajouter les méthodes qui ont du sens au niveau métier (pas les getters/setters).
  5. Vérifier qu'aucune classe ne fait double emploi (signe de mauvaise normalisation à venir).

05 Passage classe → table (aperçu, détaillé en partie 2)

Règle de passage 1 classe = 1 table · 1 attribut = 1 colonne · association 1–N = clé étrangère côté N · association N–N = table de jonction avec deux clés étrangères.

06 Exemple — PlantUML

@startuml
class Etudiant {
  -id: int
  -nom: string
  -email: string
  +sInscrire(cours: Cours)
}
class Cours {
  -id: int
  -titre: string
  -credits: int
}
class Enseignant {
  -id: int
  -nom: string
}
Etudiant "1..*" -- "1..*" Cours : suit
Enseignant "1" -- "0..*" Cours : enseigne
@enduml
